**Confidence:** HIGH

## Summary

Phase 2 implements the offline operation capability that is the core value of this entire migration. There are two independent tracks: (1) container image embedding using a two-phase build-time/runtime pattern, and (2) manifest auto-deployment using MicroShift's native kustomizePaths mechanism. Both are well-specified in MICROSHIFT_MIGRATION.md, which was authored with direct reference to the upstream microshift-io/microshift `embed_images.sh` implementation.

The image embedding pattern stores container images as raw OCI blobs (`dir:` format) in `/usr/lib/containers/storage/` at build time — this path is in the immutable bootc image layer and survives OS updates. A generated script (`/usr/bin/microshift-copy-images`) runs as `ExecStartPre` on every boot to copy from the immutable store into CRI-O's live `/var` storage. This two-phase approach is the upstream-blessed pattern and is significantly more robust than the current K3s approach of using `podman pull --root` into a secondary store.

The manifest deployment replaces the custom `observability-deploy.service` + `deploy-observability.sh` pattern with MicroShift's native kustomize auto-deployment. By adding `manifests.d/*` to `kustomizePaths` and placing kustomize-structured directories under `/usr/lib/microshift/manifests.d/`, MicroShift applies all manifests automatically on every start — no custom wait-for-API logic needed.

**Primary recommendation:** Implement the three plans in sequence: (1) build-time embedding script for MicroShift system images, (2) runtime copy script + systemd drop-in + edgeworks app image embedding, (3) kustomizePaths config update + manifests.d placement + removal of custom deploy services. Each plan has clear file-level deliverables from MICROSHIFT_MIGRATION.md.

## Standard Stack

### Core Tools

| Tool | Version | Purpose | Why Standard |
|------|---------|---------|--------------|
| `skopeo` | RPM from Fedora repos | Copy container images between transports (docker://, dir:, containers-storage:) | The only tool that handles `dir:` format natively; required by upstream embed_images.sh pattern |
| `jq` | RPM from Fedora repos | Parse release JSON manifests to extract image refs | Already in Containerfile base install; `jq -r '.images[]? // empty'` is the exact query needed |
| `sha256sum` | coreutils (present by default) | Deterministic directory naming for embedded images | `echo -n "${img}" | sha256sum | awk '{print $1}'` — filesystem-safe, collision-free |
| MicroShift kustomize | Built into MicroShift binary | Auto-apply kustomization.yaml on every start | Native mechanism — no kubectl/oc or external deploy tooling needed |

### Supporting

| Tool | Version | Purpose | When to Use |
|------|---------|---------|-------------|
| `find` + bash | coreutils | Scan `/usr/share/microshift/release/` for JSON files | Used in embed script to discover all release manifests |
| systemd drop-ins | systemd | Extend microshift.service without modifying the RPM-provided unit | Drop-in at `microshift.service.d/` — do NOT edit the RPM's unit file directly |

### Alternatives Considered

| Instead of | Could Use | Tradeoff |
|------------|-----------|----------|
| `skopeo copy dir:` format | `podman pull --root` to secondary store | Current approach (K3s era). Less robust: secondary stores require explicit configuration, may not survive bootc updates. Dir: format is upstream-standard |
| Generated `/usr/bin/microshift-copy-images` | Static script file committed to repo | Both work; generated approach keeps embedding logic in one place. Static file approach is simpler for debugging. Either is acceptable |
| `embed-microshift-images.sh` processing both MicroShift + app images | Two separate scripts | Single script is simpler. Separation is cleaner if app images need different auth. MICROSHIFT_MIGRATION.md recommends single script with app image list section appended |

### Pattern 1: Two-Phase Image Embedding

**What:** Build-time skopeo copies images to `/usr/lib/containers/storage/` as `dir:` blobs. Runtime script copies to CRI-O's `/var` storage via `ExecStartPre`.

**When to use:** Any container image that must be available offline on bootc-managed edge devices.

**Why two phases:** `/usr/lib` is the immutable bootc image layer — it survives `bootc upgrade`. `/var/lib/containers/storage` (CRI-O's live store) is mutable and may be wiped. The `ExecStartPre` guarantees images are repopulated into CRI-O before MicroShift starts on every boot, including after OS updates.

**Build-time (in Containerfile):**
```dockerfile
# Required: UID mapping for skopeo during build
RUN echo "root:1:65536" >> /etc/subuid && \
    echo "root:1:65536" >> /etc/subgid

COPY configs/edgeworks-images.txt /usr/share/edgeworks/image-list.txt
COPY scripts/embed-microshift-images.sh /tmp/embed-microshift-images.sh

RUN chmod +x /tmp/embed-microshift-images.sh && \
    if [ "${EMBED_CONTAINER_IMAGES}" = "1" ]; then \
        /tmp/embed-microshift-images.sh; \
    fi && \
    rm -f /tmp/embed-microshift-images.sh
```

**Runtime (generated by embed script, placed at build time):**
```bash
# /usr/bin/microshift-copy-images (generated by embed-microshift-images.sh)
# Runs as ExecStartPre before microshift.service
while IFS=',' read -r img sha; do
    skopeo copy --preserve-digests \
        "dir:/usr/lib/containers/storage/${sha}" \
        "containers-storage:${img}" 2>/dev/null
done < /usr/lib/containers/storage/image-list.txt
```

**Systemd drop-in (generated by embed script):**
```ini
# /usr/lib/systemd/system/microshift.service.d/microshift-copy-images.conf
[Service]
ExecStartPre=/usr/bin/microshift-copy-images
```

### Pattern 2: MicroShift kustomizePaths Manifest Auto-Deployment

**What:** MicroShift scans configured `kustomizePaths` on every start and runs `kubectl apply -k` on each directory containing a `kustomization.yaml`.

**When to use:** Any workload that should be deployed automatically on every boot (system services, application pods). Replaces custom systemd `oneshot` + `kubectl apply` patterns.

**config.yaml update:**
```yaml
manifests:
  kustomizePaths:
    - /usr/lib/microshift/manifests
    - /usr/lib/microshift/manifests.d/*
    - /etc/microshift/manifests
    - /etc/microshift/manifests.d/*
```

**Directory structure for manifests.d:**
```
/usr/lib/microshift/manifests.d/
├── 05-observability/
│   ├── kustomization.yaml
│   └── observability-stack.yaml
└── (additional numbered dirs for edgeworks app manifests)
```

Each directory must contain a `kustomization.yaml`. Directories are applied in alphabetical order — use numeric prefixes to control dependency order (infrastructure before applications).

**Minimal kustomization.yaml for plain YAML manifests:**
```yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
resources:
  - observability-stack.yaml
```

### Anti-Patterns to Avoid

- **Editing the RPM-provided microshift.service unit:** Use systemd drop-ins at `/usr/lib/systemd/system/microshift.service.d/` instead. The RPM's postinstall manages the base unit.
- **Using `podman pull --root` into a secondary store:** The `additionalImageStores` approach (current K3s pattern with `storage.conf.d/offline.conf`) is less robust across bootc updates. Use the `dir:` + `containers-storage:` skopeo pattern instead.
- **Placing manifests in `/etc/microshift/manifests.d/`:** `/etc/` is for site-specific runtime overrides. Manifests baked into the bootc image belong in `/usr/lib/microshift/manifests.d/` (immutable layer).
- **Using `:latest` tags in edgeworks-images.txt:** The embedded image hash at build time may not match what the kustomize manifest resolves at runtime. Pin to specific digests or version tags.
- **Forgetting `/etc/subuid` and `/etc/subgid` root mappings:** skopeo requires UID mapping even during container builds. Without `echo "root:1:65536"` in both files, skopeo copy will fail with permission errors.
- **Using `--all` flag in runtime copy script:** `skopeo copy --all` copies multi-arch manifests. At build time `--all` is correct (preserves all architectures in the dir). At runtime, `--all` is unnecessary and slower — but is acceptable. The upstream pattern uses `--all` at build time only.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Manifest deployment timing | Custom systemd oneshot waiting for API readiness | MicroShift native kustomizePaths | MicroShift already waits for API server internally before applying manifests. Custom wait-for-API logic is redundant and fragile |
| Image format conversion | Custom tar/oci conversion scripts | `skopeo copy docker:// dir:` and `skopeo copy dir: containers-storage:` | skopeo handles format translation, multi-arch manifests, digest preservation transparently |
| Image deduplication | Track which images already exist before copy | Check for `${DEST}/manifest.json` existence | The embed script already implements this cache check — don't skip it in the implementation |
| Kustomize path ordering | Sort/prioritize manifest application manually | Numeric directory prefixes (05-, 10-, 20-) | MicroShift applies manifests.d directories alphabetically — numbering gives explicit control |

**Key insight:** MicroShift's kustomizePaths mechanism was designed specifically for this use case (embedded workloads that auto-deploy on boot). Using it eliminates an entire category of timing bugs that afflict custom deploy services.

## Common Pitfalls

### Pitfall 1: Missing subuid/subgid Mapping Breaks skopeo During Build

**What goes wrong:** The Containerfile `RUN` for embed-microshift-images.sh fails with a permission error like `lchown ... operation not permitted` or `Error during unshare(CLONE_NEWUSER)`.

**Why it happens:** skopeo uses user namespaces for rootless operation even when running as root inside a container build. Without `/etc/subuid` and `/etc/subgid` entries for root, the UID remapping for unshare fails.

**How to avoid:** Add this RUN layer BEFORE the embed script is invoked:
```dockerfile
RUN echo "root:1:65536" >> /etc/subuid && \
    echo "root:1:65536" >> /etc/subgid
```

**Warning signs:** Build fails on the embed script RUN step with "lchown" or "unshare" errors.

### Pitfall 2: Release JSON Structure — `.images[]` vs Other Keys

**What goes wrong:** `jq -r '.images[]? // empty'` returns nothing, so no images are embedded.

**Why it happens:** The actual JSON structure in `/usr/share/microshift/release/*.json` from the COPR packages may differ slightly from what the embed script assumes. The upstream embed_images.sh uses `.images[]?` but the actual key depends on the RPM version.

**How to avoid:** After Phase 1 (Foundation) installs the COPR packages, inspect the actual release JSON structure before finalizing the embed script:
```bash
jq 'keys' /usr/share/microshift/release/*.json
```
Adjust the jq query if the structure uses a different top-level key.

**Warning signs:** Embed script reports "0 pulled, 0 skipped" but no error.

### Pitfall 3: `/usr/lib/containers/storage/` vs `/usr/share/containers/storage/`

**What goes wrong:** Images embedded to the wrong path don't survive bootc updates, or CRI-O can't find them.

**Why it happens:** The current `Containerfile.fedora.optimized` uses `/usr/share/containers/storage` with `additionalImageStores`. Phase 2 switches to `/usr/lib/containers/storage` with the two-phase copy pattern. Mixing the two approaches or using the wrong path breaks the offline guarantee.

**How to avoid:** Use `/usr/lib/containers/storage/` exclusively in embed-microshift-images.sh. Remove `storage.conf.d/offline.conf` if it exists. The runtime copy populates CRI-O's standard storage — no additional store configuration needed.

**Warning signs:** Images visible in `skopeo inspect dir:/usr/lib/...` but CRI-O reports image not found on boot.

### Pitfall 4: kustomization.yaml Relative Paths Broken When Embedded

**What goes wrong:** MicroShift fails to apply manifests with "no such file or directory" for base resources.

**Why it happens:** The edgeworks-deploy repo overlays use `../../base` relative paths. When both overlay and base are copied flat into `manifests.d/XX-component/`, the relative paths break.

**How to avoid:** Either (a) use `sed -i 's|../../base|./base|g'` in a Containerfile RUN after COPY, or (b) create dedicated `overlays/embedded/` overlays in edgeworks-deploy that use flat paths. Option (b) is cleaner long-term.

For the observability manifest (MNFT-01, current Phase 2 scope), this is not an issue — `os/manifests/observability-stack.yaml` is a plain YAML file, not a kustomize overlay with relative base references. Create a simple kustomization.yaml wrapping it.

**Warning signs:** MicroShift logs show `kustomize build` errors with path-related messages after boot.

### Pitfall 5: EMBED_CONTAINER_IMAGES=1 Makes Builds Very Slow

**What goes wrong:** Build takes 30-60 minutes to pull all MicroShift system images + 15 edgeworks app images over the network.

**Why it happens:** Each `skopeo copy docker://` fetches all image layers for all architectures (`--all` flag). With ~20+ system images and 15 app images this is substantial data.

**How to avoid:** This is expected behavior, not a bug. Use `EMBED_CONTAINER_IMAGES=0` for development/test builds. Only use `=1` for production image builds. Document this in Makefile comments.

**Warning signs:** N/A — this is expected. Plan for extended CI build times.

### Pitfall 6: observability-deploy.service Still Enabled After Removal

**What goes wrong:** Boot hangs or errors because systemctl tries to start a service whose ExecStart script no longer exists.

**Why it happens:** `systemctl enable` in the Containerfile bakes service symlinks into the image. If `observability-deploy.service` is still enabled but `deploy-observability.sh` is removed, the unit file references a missing binary.

**How to avoid:** In Plan 02-03, explicitly remove `observability-deploy.service` from the `systemctl enable` list in the Containerfile AND remove the COPY line for `systemd/observability-deploy.service`. Both must be removed together.

## Code Examples

Verified patterns from MICROSHIFT_MIGRATION.md (primary reference, HIGH confidence):

### Build-Time Embedding Script Core Loop

```bash
# Source: MICROSHIFT_MIGRATION.md — scripts/embed-microshift-images.sh
IMAGE_STORAGE_DIR="/usr/lib/containers/storage"
IMAGE_LIST_FILE="${IMAGE_STORAGE_DIR}/image-list.txt"
RELEASE_DIR="/usr/share/microshift/release"

mkdir -p "${IMAGE_STORAGE_DIR}"
: > "${IMAGE_LIST_FILE}"

for manifest in $(find "${RELEASE_DIR}" -name '*.json' -type f 2>/dev/null); do
    while IFS= read -r img; do
        [ -z "${img}" ] && continue
        # Skip Red Hat registry images (not available upstream)
        if echo "${img}" | grep -q "registry.redhat.io"; then
            continue
        fi
        SHA=$(echo -n "${img}" | sha256sum | awk '{print $1}')
        DEST="${IMAGE_STORAGE_DIR}/${SHA}"
        if [ ! -d "${DEST}" ] || [ ! -f "${DEST}/manifest.json" ]; then
            skopeo copy --all --preserve-digests "docker://${img}" "dir:${DEST}"
        fi
        echo "${img},${SHA}" >> "${IMAGE_LIST_FILE}"
    done < <(jq -r '.images[]? // empty' "${manifest}" 2>/dev/null)
done
```

### Runtime Copy Script Core Loop

```bash
# Source: MICROSHIFT_MIGRATION.md — /usr/bin/microshift-copy-images (generated)
while IFS=',' read -r img sha; do
    [ -z "${img}" ] || [ -z "${sha}" ] && continue
    SRC="/usr/lib/containers/storage/${sha}"
    if [ -d "${SRC}" ]; then
        skopeo copy --preserve-digests \
            "dir:${SRC}" "containers-storage:${img}" 2>/dev/null

done < /usr/lib/containers/storage/image-list.txt
```

## Open Questions

1. **Release JSON structure in COPR packages**
   - What we know: MICROSHIFT_MIGRATION.md specifies `jq -r '.images[]? // empty'` based on upstream embed_images.sh
   - What's unclear: Whether the nightly COPR package JSON uses `.images[]` or a different key (e.g., `.Images[]`, `.containerImages[]`)
   - Recommendation: After Phase 1 installs the COPR packages, inspect actual JSON before implementing Plan 02-01. Add a schema validation step at the top of embed-microshift-images.sh

2. **edgeworks-images.txt uses `:latest` tags**
   - What we know: `:latest` means the embedded image content may drift from what the manifest deploys
   - What's unclear: Whether edgeworks app manifests pin specific digests or also use `:latest`
   - Recommendation: Use `:latest` for now (as specified in REQUIREMENTS.md OFFL-04). Document that pinning to digests is a future improvement. Flag in edgeworks-images.txt with a comment.

3. **Harbor registry authentication during build**
   - What we know: 15 app images are on `harbor.theedgeworks.ai/edgeworks/` — a private registry
   - What's unclear: How the build environment authenticates to harbor. The embed script's `skopeo copy docker://harbor.theedgeworks.ai/...` will fail if credentials aren't available
   - Recommendation: Verify build environment has `~/.docker/config.json` or equivalent credentials for harbor before Plan 02-02. If CI handles this, ensure the CI runner has registry credentials configured.

4. **edgeworks-deploy manifests not yet in os-builder repo**
   - What we know: MNFT-01 requires manifests in `manifests.d/10-edgeworks` through `40-opcua-adapter`, but these come from the edgeworks-deploy repo
   - What's unclear: Whether the edgeworks-deploy repo content is accessible during Phase 2 planning/execution, and whether overlay kustomization.yaml files need path rewriting
   - Recommendation: Plan 02-03 scope is only `05-observability` (the existing `observability-stack.yaml`). The edgeworks-deploy manifests (10-40) may be a separate sub-plan or addressed after Phase 2. The ROADMAP shows 02-03 is one plan — treat the edgeworks-deploy manifests as included but flag the path-rewriting requirement.
